# v1.1.1 改造指导文件

## 课堂实时兜底助手改造 Todo List（面向编程型 AI 的执行版）

本文基于当前 `v1.1` 仓库结构整理，面向 `Codex`、`Claude` 等编程类 AI 使用。目标不是产品汇报式描述，而是给后续自动化改码代理提供一份可以直接执行、拆解、排序和落地的指导清单。

适用范围：

- 项目根目录：`/Users/hooked4st/v1.1`
- 当前后端：`FastAPI`
- 当前前端：`React + Vite + Tauri`
- 当前主链路：`ASR -> monitor_service -> WebSocket/question_router -> LLMService -> FallbackPanel`

面向 AI 执行时的理解方式：

- 优先把本文当成“实施约束 + 任务拆解文档”，不是讨论稿
- 如果文档与仓库现状冲突，以仓库现状和本文中的“当前项目现状”章节为准
- 如果要生成具体修改计划，应优先遵守本文中的“推荐实施顺序”和“最高优先级清单”
- 除非本文明确要求，否则不要顺手重构无关模块，不要扩大发散范围

## 零、硬性限制

以下限制对后续 Codex / Claude 等编程类 AI 属于强约束：

1. 不要大改 UI。
2. 尽可能保留原始 UI 结构、布局、交互路径和视觉风格。
3. 本轮改造以“内核替换”为主，即优先替换推理链路、配置、状态管理、健康检查、上下文机制，不以重做前端界面为目标。
4. 前端改动应以最小侵入方式完成，只允许为了接入本地模型状态、预热、重置上下文等必要能力做小范围补充。
5. 不要新增一套平行 UI，不要在现有 `FallbackPanel`、`ClassStatusPanel`、`ToolBar` 之外再造同类主面板。
6. 不要为了接入 Ollama 顺手做视觉翻新、组件体系重写、样式体系重构。
7. 旧 UI 里只要能承载新能力，优先复用，不以“更现代”“更好看”为理由改外表。
8. 如果某项需求必须改 UI，也应优先做字段补充、状态提示、按钮补充，不做整体布局重做。
9. 本轮不要实做“实时总结”本地推理链路，只保留配置开关、状态字段和后续可接入的接口。
10. 默认按本地算力有限处理，优先保障“学生提问短答”和“课后总体总结”，不要把算力优先消耗在实时总结上。

对 AI 的执行偏好：

- 优先改 `service / router / config / state / prompt builder`
- 次优先改 `api.ts` 和局部状态透传
- 最后才改组件展示层
- 任何前端改动都应回答一个问题：是不是为了支撑内核改造所必需

本轮模型分工是硬约束，不要擅自改成单模型通吃：

- 一句话兜底答案默认模型：`qwen2.5:1.5b`
- 课堂追问、短答、口语化补充默认模型：`qwen2.5:1.5b`
- 课后总体总结默认模型：`gemma4:e4b`
- 实时总结功能：默认关闭，本轮不落地本地总结生成，只保留接口和开关

## 零点五、分批执行与交接协议

由于代理上下文窗口有限，本文不能按“一次性完成全部改造”来执行。后续 Codex / Claude 等编程型 AI 必须按“小任务包”推进，每次只完成一小批明确目标，并在结束前回写进度，供下一个 agent 接力。

### 执行原则

1. 每次只做一个小任务包，禁止一次跨越多个大阶段。
2. 每次任务包以 `1` 到 `3` 个主任务为上限，且这些任务必须强相关。
3. 如果某项任务会牵连大量前后端和文档联动，必须继续向下拆，不要一次吃完。
4. 每次结束时，必须更新本文中的完成标记，不允许只改代码不回写 Todo。
5. 每次结束时，必须补一段“交接记录”，告诉下一个 agent：
   - 本次完成了什么
   - 改了哪些文件
   - 哪些任务已经完成
   - 哪些任务完成了一半
   - 下一次最适合继续做哪几项
6. 如果任务未完成，不要勾选完成；可在交接记录中注明“进行到哪一步”。

### 勾选规则

- `[ ]` 表示未开始
- `[x]` 表示本项已完成并已落盘验证
- `[~]` 表示部分完成，但还不能算完成

禁止行为：

- 不要因为只创建了文件骨架就把整项标成 `[x]`
- 不要因为只改了后端未改前端，就把“前后端联动项”标成 `[x]`
- 不要跳过回写，让下一个 agent 自己猜进度

### 每次结束时必须追加的交接记录格式

文档末尾维护一个“执行批次记录”区域。每完成一轮任务，必须追加一条新记录，格式统一如下：

```md
### Batch N

- 日期：YYYY-MM-DD
- 执行者：Codex / Claude / 其他 agent 名称
- 本次目标：
  - 任务 A
  - 任务 B
- 实际完成：
  - 已完成项 1
  - 已完成项 2
- 修改文件：
  - path/a
  - path/b
- Todo 状态变更：
  - `[x]` 第 X 项
  - `[~]` 第 Y 项
- 遗留问题：
  - 问题 1
  - 问题 2
- 建议下次继续：
  - 下一批先做什么
```

### 推荐单次任务大小

适合一次完成的小任务：

- 只改后端配置和 `.env` 映射
- 只接 Ollama client 和 health 接口
- 只替换问答主链路
- 只补设置面板里的本地模型字段
- 只补课后总结模型切换
- 只补实时总结开关和空接口

不适合一次完成的大任务：

- 同时重写后端推理、前端状态、上下文管理、日志、测试、文档
- 同时做“短答链路 + 课后总结 + 实时总结 + UI 重构”
- 一次性从第 5 项改到第 20 项

## 一、先看当前项目现状

在开始改造前，先统一对当前仓库的判断，后续开发都以这部分为事实基础：

- 当前云端 LLM 主入口集中在 `backend/services/llm_service.py`
- `question_router.py` 已是教师版主问答链路，`rescue_router.py` 仍有旧救场残留
- `summary_service.py`、`monitor_service.py` 都直接依赖 `LLMService()`
- `.env` 设置入口已经存在，后端通过 `backend/routers/settings_router.py` 读写，前端通过 `frontend/src/components/SettingsPanel.tsx` 编辑
- 上课开始入口已存在：`frontend/src/components/StartMonitorPanel.tsx` -> `frontend/src/services/api.ts` -> `backend/routers/monitor_router.py`
- 当前会话状态核心仍偏轻量，主要散落在 `backend/services/monitor_service.py` 与 `backend/services/transcript_service.py`
- 当前前端已有课堂状态展示能力，但还没有“本地模型状态 / 预热状态 / 重置上下文”的专门 UI
- 当前打包配置仍包含旧模块痕迹，`backend/backend.spec` 仍把 `routers.rescue_router` 打进包

结论：

1. 这个项目不是从零开始接 Ollama，而是要把现有 `LLMService` 为核心的云端调用链，系统性迁到“本地推理优先”。
2. 这次改造不能只改一个 service，必须同时覆盖后端服务、环境配置、前端设置面板、开始课堂流程、打包配置、文档和测试。
3. 后续开发应优先围绕当前已经在使用的新教师链路改造，避免继续给 `rescue_router.py` / `RescuePanel.tsx` 这类旧残留加功能。

---

## 二、改造总原则

- 云端只保留字节 ASR；课堂文本不再发往云端 LLM
- 本地推理优先；所有生成链路默认走 Ollama
- 先替换链路，再做提示词和上下文精调
- 先收敛旧入口，再扩展新能力
- 先让首答稳定，再追求更复杂的追问、多模板和评估

建议按以下顺序推进：

1. 架构与残留收敛
2. 本地 Ollama 接入
3. 配置与启动流程打通
4. 提示词、科目、问题类型改造
5. 上下文压缩与重置机制
6. HUD 和交互补齐
7. 日志、评估、模型对比
8. 文档和迁移说明补全
