# v1.1 开发者文档（含 v1.1.1 改造约束）

本文件是 `v1.1` 仓库的开发准则。

如果旧项目现状、旧 README、旧命名方式与本文件冲突，一律以当前仓库现状和根目录 [v1.1.1.md](v1.1.1.md) 为准。

## v1.1.1 当前执行基线

`v1.1.1` 阶段的目标是把课堂主链路从“云端 LLM 优先”迁到“本地推理优先”，同时尽量不重做现有 UI。

当前执行时必须遵守以下边界：

- 云端只保留 ASR，不再把课堂文本默认发往云端 LLM
- 本地 `Ollama` 负责短答、追问、课后总结等推理能力
- 前端负责 HUD、状态展示和老师操作入口，不新增平行主面板
- 后端负责问题检测、上下文管理、本地模型调用和结果回传

当前教师版主链路以以下文件为准：

- `backend/services/llm_service.py`
- `backend/services/monitor_service.py`
- `backend/routers/question_router.py`
- `backend/routers/monitor_router.py`
- `frontend/src/components/FallbackPanel.tsx`
- `frontend/src/components/ClassStatusPanel.tsx`
- `frontend/src/App.tsx`

以下文件是历史兼容残留，不再作为新功能默认落点：

- `backend/routers/rescue_router.py`
- `frontend/src/components/RescuePanel.tsx`
- `backend/backend.spec` 中的 `routers.rescue_router`

## v1.1.1 本地推理交付约定

收尾阶段默认还要同时维护三类交付物：

- 运行时依赖：`backend/requirements.txt`
- 打包入口：`backend/backend.spec`、`backend/class-assistant-backend.spec`
- 迁移文档：`README.md`、`developer-guide.md`、`docs/local-llm-migration.md`

具体要求：

- 新增 `backend/services/*` 或 `backend/routers/*` 模块后，要同步确认两个 `.spec` 能覆盖这些模块
- `.env.example` 需要随打包产物保留，便于部署后校对 `OLLAMA_*` 配置
- 文档必须明确区分“本地主链路”和“历史兼容链路”，不要再把 `LLM_*` 写成默认前提
- 课前准备、模型拉取、故障排查统一以 `docs/local-llm-migration.md` 为准

## 1. 项目定义

### 1.1 一句话定位

`v1.1` 是一个在上课过程中帮助老师实时兜底答疑的 Mac 桌面工具。

### 1.2 用户与场景

- 用户：老师
- 场景：一对一 / 一对二课堂
- 核心触发：学生提出明确问题
- 核心输出：一句话兜底答案 + 可展开知识卡片

### 1.3 非目标

- 不是课堂应付型工具
- 不是高干扰告警工具
- 不是规避课堂互动的辅助工具
- 不是单纯会议记录工具

## 2. 开发总原则

### 2.1 先重构骨架，再碰产品核心

- Phase 0-2 先做目录、迁移、语义清洗
- Phase 3 之后才进入老师模式 MVP
- 不允许把旧 legacy 课堂应急逻辑直接包装成新产品上线

### 2.2 先保留底层能力，再替换产品语义

- 可复用：Tauri、FastAPI、ASR、WebSocket、滚动摘要、资料解析、数据落盘
- 必须重写：角色语义、主触发链路、Prompt、主界面、课后产物

### 2.3 开发判断优先级

每次改动前先问 4 个问题：

1. 这次改动是否让产品更像老师工具，而不是旧课堂应急工具？
2. 这次改动是否服务“学生提问 -> 老师得到兜底答案”主链路？
3. 这次改动是否会污染旧项目，或继续依赖旧路径？
4. 这次改动是否适合当前阶段，还是超前开发？

如果上述任一答案明显不合理，暂停改动，先回到本文件重新校准。

## 3. 目录约定

`v1.1` 目录按以下结构维护：

```text
v1.1/
├── frontend/          # 新前端与 Tauri 桌面壳
├── backend/           # 新后端与服务骨架
├── docs/              # 开发主文档、定位、迁移表、术语表
├── data/
│   ├── transcripts/   # 实时转写
│   ├── summaries/     # 课后总结
│   ├── materials/     # 当前课堂资料与解析结果
│   ├── timeline/      # 问题时间轴与书签
│   └── debug/         # 调试输出
├── prompts/
│   └── legacy/        # legacy prompt 冻结区
└── scripts/           # 启动、构建、迁移、调试脚本
```

规则：

- `v1.1` 是独立工作区，不直接写入旧 `data/`
- 新逻辑只允许依赖 `v1.1` 内部路径
- 旧项目代码只作为参考底盘，不再作为正式实现边界

## 5. 阶段路线图

## Phase 0：项目目录重构

### 目标

先立新骨架，不碰核心逻辑。

### 必做

- 建立 `v1.0` 独立工作区
- 补齐基础目录
- 明确旧项目是参考底盘，新项目是正式重构项目
- 输出产品定位、迁移映射、术语替换 3 份基础文档

### 不做

- 不改旧项目业务逻辑
- 不直接搬 UI 改样式
- 不提前做老师模式功能

### 完成标准

- `v1.0` 基础结构完整
- 有文档可约束后续开发
- 原项目文件保持不动

## Phase 1：底盘迁移

### 目标

只搬轮子，不改产品方向。

### 必做

- 将旧前端/Tauri 壳迁入 `frontend/`
- 将旧 FastAPI 骨架迁入 `backend/`
- 迁移配置模板与独立数据目录
- 确认前后端都能各自启动一次

### 不做

- 不重做 UI
- 不接新接口
- 不重写核心业务
- 不保留任何旧路径硬编码

### 完成标准

- `frontend` 可独立启动
- `backend` 可独立启动
- 运行时只写入 `v1.0/data/`

## Phase 2：去学生化

### 目标

清掉旧产品的课堂应急语义。

### 必做

- 全项目排查旧课堂应急视角文案、命名、说明文档
- 产出旧术语到新术语替换表
- 禁用 legacy 点名预警 / 关键词报警主业务链
- 冻结 legacy 应急回答 prompt 到 `prompts/legacy/`

### 不做

- 不删除底层实时监听、推送、关键词能力
- 不把 legacy prompt 混入新主流程

### 完成标准

- `v1.0` 主文案围绕老师场景
- 默认主流程不再以点名警报驱动
- legacy prompt 不再被主流程调用

## Phase 3：老师模式 MVP

### 目标

做最小闭环，而不是做大全。

### MVP 仅包含

1. 实时转写
2. 学生明确问句检测
3. 一句话兜底答案
4. 可展开卡片
5. 暂停键

### 明确排除

- 完美说话人识别
- 复杂知识图谱
- 花哨总结 UI
- 多模型高级编排
- 完整课后报告系统

### 完成标准

- 可模拟一节课
- 学生一问，系统能稳定提示

## Phase 4：问句触发链路

### 目标

把旧“关键词报警”改造成“学生提问事件”引擎。

### 必做

- 定义学生提问事件数据结构
- WebSocket 主消息改为老师语境
- 以问句触发替代点名/抽查触发
- 增加低干扰触发门槛

### 第一版门槛

- 相似问题短时间不重复提示
- 非明确问句不提示
- 一次只展示一个主提示
- 老师未处理前不叠多个弹窗

### 完成标准

- 后端可记录“学生提问事件”
- WebSocket 可推送问句相关事件
- 提示克制，不打断课堂

## Phase 5：兜底答案生成

### 目标

把旧 legacy 应急回答逻辑替换成老师可直接讲的答案。

### 必做

- 定义默认输出格式
- 重写老师兜底 Prompt
- 建立受控上下文压缩输入

### 默认输出格式

- 默认显示：一句话兜底答案
- 展开显示：定义/结论、原因/条件、例子/易错点

### 输出约束

- 短
- 稳
- 可直接说出口
- 不长篇解释
- 不输出大量无关信息

### 完成标准

- 回答 10 秒内可扫读
- 语言像老师备课卡片，不像临场应付话术

## Phase 6：卡片与交互

### 目标

让界面看起来就是老师的课堂 HUD。

### 必做

- 一句话兜底卡片
- 展开式知识卡片
- 暂停键与可见状态

### 主界面必须有

- 当前状态区
- 当前学科模式
- 最近识别问题
- 一句话兜底卡片
- 展开按钮
- 暂停按钮

### 主界面必须没有

- 红色惊慌覆盖层
- legacy 应急按钮
- 旧课堂进度入口
- 过多聊天式追问区

### 完成标准

- 老师讲课时能快速扫读
- 界面语义明确属于老师工具

## Phase 7：课堂状态压缩

### 目标

把普通摘要升级成可持续驱动回答的课堂状态。

### 必做

- 定义课堂状态结构
- 建立滚动摘要更新机制
- 建立已解释概念缓存

### 课堂状态至少包含

- 当前学科
- 当前话题
- 已解释概念
- 当前学生卡点
- 最近关键提问
- 当前引用资料

### 完成标准

- 上下文不会无限增长
- 回答能利用课堂状态，而不是重复猜

## Phase 8：时间轴与书签

### 目标

让每次关键问答可回看、可复盘。

### 必做

- 设计问题时间轴节点
- 自动加入提问书签

### 自动入书签候选条件

- 明确问句
- 已生成兜底答案
- 老师点击过展开
- 同一问题被重复追问

### 完成标准

- 下课后能快速看到关键问题轨迹
- 书签不是流水账

## Phase 9：课后总结重构

### 目标

总结服务老师备课，不服务课后补课式场景。

### 必做

- 重写课后总结结构
- 区分课堂总结与知识入库卡片

### 新总结至少包含

1. 本节课知识点
2. 学生提问点
3. 学生不会的点
4. 关键时间轴问题
5. 下节课建议提前讲的内容

### 完成标准

- 总结可直接支撑下一节课备课

## Phase 10：资料引用与知识库适配

### 目标

把资料引用改造成老师备课辅助层。

### 必做

- 保留 PDF/Word/PPT/讲义解析
- 建立当前课堂资料机制
- 回答时只命中相关片段，不整份灌入

### 完成标准

- 资料引用辅助答疑，而不是堆 token
- 系统知道本节课正在围绕哪份资料回答

## Phase 11：Mac 平台化

### 目标

让 `v1.0` 成为真正以 macOS 为中心的项目。

### 必做

- 排查 `bat`、`ps1`、`exe`、PowerShell、`taskkill`
- 清理 Windows 专属启动流程
- 确认音频采集与启动停止逻辑适配 macOS
- 补齐 Mac 开发文档

### 完成标准

- 启动、停止、调试流程都以 Mac 为中心
- 几天后回来看仍可无痛继续开发

## Phase 12：调试与验收

### 目标

用模拟课堂验证主链路，而不是只看单点功能。

### 必做测试

- 模拟学生提问
- 模拟连续追问
- 模拟无关闲聊
- 模拟老师暂停系统
- 模拟长时间课堂

### 最低验收标准

- 新项目可独立启动
- 能实时转写
- 能识别显性学生问句
- 能弹出一句话兜底答案
- 能展开查看
- 能暂停
- 能记录问题时间轴
- 课后能输出知识点与学生提问点

## 6. 开发禁令

以下内容在对应阶段前不要提前做：

- 不提前做复杂知识图谱
- 不提前做完整课后报告系统
- 不提前做高级多模型编排
- 不提前为了“准确率完美”把系统做重
- 不把旧 legacy 逻辑简单改名后继续沿用

## 7. 每次开发的执行模板

后续每次进入 `v1.0` 开发，默认按下面顺序执行：

1. 先读本文件与产品定位文档
2. 判断当前任务属于哪个 Phase
3. 明确本次改动的“必做 / 不做 / 完成标准”
4. 若涉及旧模块迁移，先查迁移映射表
5. 若涉及命名或文案，先查术语替换表
6. 完成后用本阶段验收标准自测

## 8. 当前结论

`v1.0` 的正确方向不是“更聪明的课堂应付工具”，而是“低干扰、能实时帮老师兜底答疑的课堂工具”。
